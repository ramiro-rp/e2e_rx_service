# Training and Model Documentation

## Purpose

This document describes the training-side part of the project: the dataset basis, model architecture, notebook role, exported artifacts, and the relationship between the notebook and the deployed service.

The objective is to keep a clear separation between:

- **training and artifact generation** in the notebook,
- and **runtime inference behavior** in the backend service.

## 1. Notebook role

The final notebook is responsible for:

- data preparation,
- training,
- evaluation,
- calibration,
- threshold selection,
- artifact export,
- and local visual sanity-checking.

It should not be treated as the full end-to-end application.

### Canonical notebook file
Recommended final notebook path:

```text
notebooks/rsna_training_pipeline.ipynb
```

## 2. Dataset basis

The training workflow is based on the **RSNA Pneumonia Detection Challenge** dataset from Kaggle.

- Official competition page: `https://www.kaggle.com/competitions/rsna-pneumonia-detection-challenge`

### Short description
This dataset provides chest X-ray studies and associated labels/annotations used as the practical basis for pneumonia-related training and evaluation in this project.

### Repository note
The dataset is **not included** in this repository and must be obtained separately by the user.

## 3. Clinical task

The trained model addresses a binary chest X-ray decision task:

- **positive** -> findings compatible with pneumonia / lung opacity
- **negative** -> no positive decision for the target class

This is the same semantic meaning used later by the backend and frontend.

## 4. Model architecture

The final trained model is based on:

- **ResNet-18**
- internal project-level identifier:
  - `resnet18_rsna`

In simple terms, the project uses a convolutional neural network based on ResNet-18 for binary chest X-ray classification.

## 5. Input representation

The model performs inference on a resized representation of the image.

### Key point
The model input size is:

- `224 x 224`

The original study dimensions are preserved separately for traceability and later visualization, but the model itself predicts from the resized representation.

## 6. Data handling role of the notebook

The notebook is expected to contain:
- dataset preparation logic,
- train/validation/test preparation,
- transformations and preprocessing,
- and final experiment export.

If the dataset is not committed, the notebook should still make clear:
- what dataset is expected,
- how it is locally organized,
- and what artifacts are produced.

## 7. Evaluation and calibration

The notebook should include:
- validation/testing metrics,
- temperature scaling or equivalent calibration,
- threshold selection,
- and export of the final operational threshold used by the service.

### Important distinction
The threshold is not a metric. It is the decision cutoff used to transform calibrated probability into a class label.

## 8. Exported artifacts used by the backend

The backend consumes artifacts generated by the notebook.

### Required artifacts
Typical required files:

- `best_model.pt`
- `temperature.json`
- `threshold.json`

### Supporting artifacts
Optional additional files may include:
- `calibration_metrics.json`
- summary metric files
- figures for internal review

These artifacts allow the runtime backend to stay lightweight and artifact-driven.

## 9. What passes from notebook to service

The boundary between notebook and service should be clear.

### Notebook outputs that go to the service
- final model checkpoint
- calibration temperature
- threshold configuration
- supporting metric files when needed

### What stays in the notebook
- experiment logic
- training loops
- evaluation code
- local plots
- local visual sanity-checks

### What belongs to the backend instead
- API contracts
- QC response behavior
- Orthanc integration
- batch processing
- derived DICOM generation at runtime
- frontend-visible response structure

## 10. Local inference demo in the notebook

Keeping a final inference/demo section in the notebook is acceptable.

Its purpose should be:
- to confirm that the exported model works,
- to visually inspect predictions,
- to validate Grad-CAM behavior locally.

It should not be presented as the deployed runtime path.

## 11. Relationship to explainability outputs

The notebook may generate:
- model-space heatmaps,
- overlays,
- or local visual examples.

The deployed service later reuses the same general interpretability logic, but the notebook remains the training and validation environment, not the final service.

### Important note
Grad-CAM is an explanatory visualization, not segmentation.

## 12. Recommended naming alignment

To keep naming consistent across the repository, documentation, and service outputs, use:

- **RSNA DICOM Service MVP**

Recommended notebook filename:
- `rsna_training_pipeline.ipynb`

## 14. Current limitations

- The notebook is not intended to document the full deployment stack.
- Large model checkpoints may remain local and not be committed to GitHub.
- Standard image inference is supported in the service, but the strongest validated route remains the DICOM workflow.
- The current project is an academic technical MVP, not a clinically validated product.

## 15. Summary

The notebook is the **training and artifact-generation layer** of the project.

Its final deliverable to the service is not the whole pipeline, but the set of trained artifacts and validated settings required by the runtime backend.
